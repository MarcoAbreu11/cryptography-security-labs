# Cryptography and Security: Assignment 2

## 1. Classical Ciphers (Substitution and Polyalphabetic)

In this section we explore the fundamentals of historical cryptography, where security relied on keeping the algorithm secret or on very short keys.

### Caesar Cipher (`cesar.py` & `cesar_attack.py`)

The Caesar cipher is one of the simplest techniques, based on substituting each letter with another that is a fixed number of positions ahead in the alphabet.

* **`cesar.py`**: Implements the cipher with automatic pre-processing (conversion to uppercase and filtering of non-alphabetic characters).
* **`cesar_attack.py`**: Performs a brute-force attack testing all 26 possible keys. Since the key space is extremely small, the attack is immediate.

---

### Vigenère Cipher (`vigenere.py` & `vigenere_attack.py`)

An evolution of the Caesar cipher that uses a keyword to apply different shifts to each letter of the message.

#### **`vigenere.py`**

* **How it works**: If the key is "BACO", each letter of the message is shifted according to the corresponding letter of the key (B=1, A=0, C=2, O=14).
* **Security**: More robust than Caesar, but still vulnerable to attacks based on statistical analysis.

---

#### **`vigenere_attack.py`**

* **How it works**: The attack assumes the key length is known. The ciphertext is split into several "slices", where each slice corresponds to a Caesar cipher.
* For each slice, the most frequent letter is identified and assumed to correspond to one of the most common letters in Portuguese (A, E, O, S).
* From this assumption, the shift is calculated and the key is reconstructed.
* **Limitation**: This method is heuristic and depends on the quality of the statistical analysis, being more effective for longer texts.

## 2. Modern Cryptography: One-Time Pad (OTP)

The One-Time Pad is a cipher technique that uses a random key of the same length as the message. The core operation is **XOR (Exclusive OR)** between the bits of the message and the bits of the key.

---

### `otp.py` (Secure Implementation)

This script follows security best practices.

* **Key Generation**: Uses `os.urandom()`, a cryptographically secure generator (CSPRNG) that provides unpredictable values.
* **Operations**:
  * `setup`: generates the key
  * `enc`: encrypts the message
  * `dec`: decrypts the message

When used correctly, this guarantees **absolute security**.

---

### `my_otp.py` (Intentional Vulnerability)

This program demonstrates the risks of poor randomness generation.

* **How it works**:
  * Generates a seed with `os.urandom(16)`
  * Applies the `seed_expand` function
  * Initializes the `random` generator with that seed
  * Generates the key with `random.randbytes(n)`

* **Vulnerability**:
  * The `seed_expand` function drastically reduces the entropy of the seed
  * The effective seed space becomes small (≈ 2¹⁶)
  * The `random` generator is deterministic

* **Consequence**:
  * The key is no longer truly random
  * It becomes possible to predict or reconstruct the key

---

## 3. Brute-Force Attack (`my_otp_attack.py`)

This script exploits the vulnerability of `my_otp.py`.

### Attack logic:

1. Read the ciphertext
2. Try all possible seeds (≈ 2¹⁶)
3. For each seed:
   * Initialize `random.seed(seed)`
   * Generate the key with `random.randbytes`
   * Decrypt using XOR
4. Check whether the result contains known words
5. When found, return the original text

## Analysis and Answers to the Questions

### Q1: Differences between `otp.py` and `my_otp.py`

The main difference lies in the source of randomness.

* **`otp.py`**:
  * Uses `os.urandom`
  * Generates truly random and unpredictable values
  * Guarantees cryptographic security

* **`my_otp.py`**:
  * Uses a pseudo-random generator (`random`)
  * Depends on a seed
  * Due to the entropy reduction caused by `seed_expand`, the seed space is small
  * Brute-force attacks become feasible

---

### Q2: Absolute Security vs Practical Attack

The attack performed **does not contradict** the absolute security of the One-Time Pad.

* OTP security depends on fundamental assumptions:
  * a truly random key
  * a key the same length as the message
  * a key used only once

* In `my_otp.py`, these requirements are violated:
  * the key is generated by a PRNG
  * the seed has low entropy

Therefore, the attack does not break the OTP itself, but rather an **incorrect implementation** of it.

## Conclusion

This assignment demonstrates that the security of a cryptographic system depends not only on the algorithm, but also on the quality of key generation. An incorrect implementation can completely compromise the theoretical security of the system.
